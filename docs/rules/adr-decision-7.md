# 🎯 의사결정 / ADR 7수칙 (E1~E7) — 왜 이렇게 했는가

> 주니어는 "어떻게 짤까?", 시니어는 "**왜 이걸 선택하는가**".

**언제 읽어야 하는가**:
- 기술 스택 선택 시 (DB, 결제 PG, 인증 등)
- 아키텍처 큰 변경 시
- 외부 서비스 도입 시
- 라이브러리/프레임워크 선택 시
- "이거 왜 이렇게 했지?" 질문이 나올 만한 결정 시

---

## E1. ADR (Architecture Decision Records) 작성

- 중요한 기술 선택은 **반드시 문서화**
- 파일: `docs/adr/ADR-NNN-제목.md`

### 표준 템플릿
```markdown
# ADR-NNN: 제목

## 상태
승인됨 | 제안 | 거절 | 폐기 (YYYY-MM-DD)

## 컨텍스트
어떤 상황에서 이 결정이 필요한가?

## 결정
무엇을 선택했는가?

## 대안
- 옵션 A: (왜 거절했는지)
- 옵션 B: (왜 거절했는지)

## 결과 (파급)
긍정/부정 결과 모두 명시

## 후속 결정
이 결정으로 인해 미래에 결정해야 할 것
```

### 예시
```markdown
# ADR-001: 결제 PG로 토스 + Stripe 이중 채택

## 상태
승인됨 (2026-05-30)

## 컨텍스트
국내 + 해외 고객 동시 서비스 필요.

## 결정
- 국내: 토스페이먼츠
- 해외: Stripe

## 대안
- 토스만: 해외 카드 결제 약함
- Stripe만: 한국 카드/카카오페이 제한
- 자체 결제: PCI-DSS 부담

## 결과 (파급)
- 운영 복잡도 ↑ (PG 2개 관리)
- 회계 리포팅 분리 필요
- 환율 처리 (KRW vs USD)

## 후속 결정
- PayPal 추가 여부 (해외 일부 시장)
- 정산 통합 방법
```

## E2. 트레이드오프 명시 (Pros / Cons / Risks)

- 모든 선택에는 **공짜가 없음**
- 장점만 적지 말 것 — **단점/위험도 명시**

### 표준 형식
```
✅ 장점
- 빠른 개발 (1주)
- 검증된 솔루션

❌ 단점
- 종속성 ↑ (벤더 락인)
- 월 사용료 발생

⚠️ 위험
- 가격 인상 시 대안 없음
- 정책 변경 시 대응 어려움

📊 정량 비교
- 자체 구현: 4주 / 0원 + 유지보수 ↑
- 외부 도입: 1주 / 월 30만원 + 유지보수 ↓
```

## E3. 가역적 vs 비가역적 결정 구분 (Type 1 vs Type 2)

- **Type 1 (비가역)**: 신중히 결정
  - DB 스키마 (마이그레이션 어려움)
  - 인증 시스템 (모든 사용자 영향)
  - 결제 PG (재계약 어려움)
  - 도메인/브랜드명

- **Type 2 (가역)**: 빠르게 결정 후 학습
  - UI 색상/레이아웃
  - 카피라이팅
  - 작은 기능 추가
  - 내부 도구

### 판단 기준
- 70% 확신이면 Type 2는 진행 (Jeff Bezos)
- **Type 1을 Type 2처럼 처리하면 사고**
- "되돌리는 데 며칠 걸리나?" 자문

### 우리 프로젝트 적용
| 결정 | Type | 검토 깊이 |
|---|---|---|
| 결제 PG 선택 | Type 1 | 깊게 (ADR 필수) |
| 버튼 색상 | Type 2 | 빠르게 (A/B 테스트) |
| Supabase 채택 | Type 1 | 깊게 |
| 폰트 변경 | Type 2 | 빠르게 |

## E4. "선택하지 않은 옵션" 기록

- ADR에 거절된 대안을 **이유와 함께** 기록
- 6개월 뒤 "왜 X 안 썼지?" → ADR에 답 있음
- 새 팀원 온보딩에 가장 가치 있는 자료

### 거절 이유 형식
```markdown
## 거절된 대안

### Firebase (대신 Supabase 선택)
- ❌ Vendor lock-in (Google)
- ❌ PostgreSQL 미지원
- ❌ 한국 reg latency
- ✅ 모바일 SDK 완성도 (인정)

### 자체 백엔드 (대신 Supabase 선택)
- ❌ 운영 부담 (DB/auth/storage 모두 자체)
- ❌ 초기 개발 4주 vs 1주
- ✅ 자유도 (인정, 그러나 현재 불필요)
```

## E5. 가정 검증 (Assumption Testing)

- "사용자가 이렇게 쓸 것이다" 가정은 **반드시 검증**

### 검증 방법
- **A/B 테스트**: 1~2주 데이터 비교
- **사용자 인터뷰**: 5명이면 80% 패턴 발견
- **분석 데이터**: 기존 사용 패턴 확인
- **프로토타입**: 실제 사용해보기

### 가정만으로 큰 결정 ❌
- "사용자는 이런 UI를 좋아할 거야" → 데이터로 확인
- "이 기능 많이 쓸 거야" → MVP로 검증
- "이 가격이면 살 거야" → 사전 조사

### 우리 프로젝트 가정 사례
- "고객은 회사 마스터 키 제공 시 만족할 것" → ✅ 검증 필요 (수요 인터뷰)
- "Flux 1.1 Pro 품질이 충분할 것" → ✅ 테스트 결과 확인
- "Pro 플랜 ₩59,000이 적정" → ⚠️ A/B 가격 테스트 필요

## E6. "지금 결정 vs 나중 결정" 판단 (Last Responsible Moment)

- **지금 결정해야 할 것**: 후행 작업 막는 것
- **나중 결정**: 정보 더 모이면 좋은 것
- 너무 빠른 결정 = 정보 부족
- 너무 늦은 결정 = 진행 정체

### 자문
"이 결정을 지금 안 하면 진행 막히나?"
- Yes → 지금 결정
- No → 나중에 더 정보 모은 후

### 예시
- **지금 결정**: DB 스키마 (모든 기능 의존)
- **나중 결정**: 고급 UI 디테일, 마이너 기능 추가
- **지금 결정**: 인증 방식 (모든 페이지 영향)
- **나중 결정**: 다국어 지원 (사용자 분포 확인 후)

## E7. RFC (Request for Comments) 프로세스

- 큰 변경은 **사전 RFC 작성** → 팀 리뷰 후 결정
- 1인 결정 → 사후 비난
- RFC 후 합의 → 책임 공유

### 우리 프로젝트 적용 (마스터-Claude)
- Claude가 큰 결정 제안 시 **반드시**:
  - 옵션 2개 이상 제시
  - 각 옵션의 트레이드오프 명시
  - 권장안 + 이유 명시
  - 마스터 결정 대기

### RFC 양식 (간소화)
```markdown
# RFC: 제목

## 배경 / 문제
무엇을 해결하려는가?

## 제안
어떻게 할 것인가?

## 옵션 비교
A: ... (장단점)
B: ... (장단점)
C: ... (장단점)

## 권장
B를 권장. 이유: ...

## 미해결 질문
- ?
- ?

## 일정
- 결정: MM/DD
- 구현 시작: MM/DD
- 출시: MM/DD
```
