## H1. 기술부채 시각화 (Make Debt Visible)

머릿속에만 있으면 우선순위 못 매김.

### 도구
- `TODO(#123): ...` (이슈 번호 연결)
- 부채 대시보드 (코드 복잡도, 테스트 커버리지, 의존성 outdated)
- `docs/tech-debt.md` 백로그

### 측정 메트릭
- 함수당 평균 줄 수
- 순환 복잡도 (Cyclomatic Complexity) — 10 이상 위험
- 중복 코드 % (DRY 위반)
- 테스트 커버리지 변화 추세
- TODO/FIXME/HACK 개수

### 우리 프로젝트 부채 (현재)
- `sns-platform/`은 ES5 HTML/JS → 향후 React + TS 마이그레이션 대상
- 일부 페이지(`ai-writer.html`)는 옛 단일 계정 형식 사용 (멀티 계정 마이그 필요)
- 테스트 코드 부재 (TDD 도입 시 채우기)

## H2. ROI 평가 (Refactor vs Stay)

리팩터는 **공짜가 아님** — 비용 vs 효과.

### 매트릭스
|  | 자주 수정 | 가끔 수정 |
|---|---|---|
| **복잡한 코드** | 🔥 즉시 리팩터 | ⏳ 나중에 |
| **단순한 코드** | ✅ 두면 됨 | 💀 무시 OK |

### ROI 계산
```
효과 = (리팩터 후 수정 시간 단축) × (앞으로 수정 횟수)
비용 = 리팩터 시간 + 위험(버그 도입)

리팩터 진행: 효과 > 비용
유지: 효과 ≤ 비용
```

### 우리 프로젝트 적용 예시
- `settings.html` (자주 수정 + 복잡) → 🔥 React 마이그레이션 우선순위
- `safety-guide.html` (가끔 수정 + 단순) → 💀 그대로 두기
- `publish.html` (자주 수정 + 복잡) → 🔥 우선

**"어차피 안 건드릴 코드"는 못생겨도 OK.**

## H3. Strangler Fig 패턴 (점진 교체)

큰 시스템 통째 재작성 ❌ (대부분 실패).

### 패턴 흐름
```
1. 새 시스템 만들기 (옛것 옆에)
2. 새 기능부터 새 시스템에
3. 기존 기능 점진 이전 (월 1개씩)
4. 완전 이전 후 옛 시스템 제거
```

### 우리 사례
- ES5 HTML/JS → React + TS 마이그레이션
- 새 페이지(`billing.html` 등)부터 React로 만들기
- 기존 페이지는 그대로 동작
- 사용 빈도 높은 순으로 마이그레이션
- 완전 이전 후 ES5 제거

### 장점
- 항상 동작하는 상태 유지
- 위험 분산
- 학습 누적 (다음 마이그레이션 쉬워짐)

### 함정
- "병행 운영 비용" 과소평가 → 명시적으로 계획
- 옛것 제거 단계 잊음 → 데드라인 설정

## H4. Branch by Abstraction (안전한 리팩터)

### 단계
```
1. 기존 구현 위에 [추상화 계층] 추가
2. 새 구현 만들기 (기존과 병행)
3. 호출자를 추상화 통해 전환
4. 옛 구현 제거
```

### 예시
```ts
// Step 1: 추상화
interface PaymentProvider {
  charge(amount: number): Promise<Result>;
}

// Step 2: 기존 구현
class TossProvider implements PaymentProvider { ... }

// Step 3: 새 구현 추가
class StripeProvider implements PaymentProvider { ... }

// Step 4: 호출자 코드는 추상화만 의존
function pay(provider: PaymentProvider, amount: number) {
  return provider.charge(amount);
}

// Step 5: feature flag로 점진 전환
const provider = useStripe ? new StripeProvider() : new TossProvider();
```

중간 상태 항상 동작 → 안전한 리팩터.

## H5. 보이스카웃 룰 (Boy Scout Rule)

> "온 것보다 깨끗하게 떠나라"

- 한 번에 다 고치지 말고, **만질 때마다 조금씩**
- 매번 함수 하나 정리 → 1년 누적 = 큰 개선
- **관련 없는 리팩터 금지** (안전수칙 #3)

### 적절한 보이스카웃 활동
- 변수명 더 명확하게 (한 변수)
- 매직넘버 → 상수 (그 함수 안만)
- 너무 긴 함수 → 작게 분리 (작업 중인 함수)
- 명백한 버그 수정 (작업 중 발견)

### 금지되는 활동 (수칙 #3 위반)
- 전체 폴더 구조 재편
- 라이브러리 교체
- 새 기능 추가
- 광범위 포맷팅

## H6. 리팩터 vs 재작성 판단

### 리팩터 선택 조건
- ✅ 동작은 OK, 구조만 문제
- ✅ 테스트 있음 (안전망)
- ✅ 점진 가능
- ✅ 비즈니스 로직 명확

### 재작성 선택 조건
- ❌ 동작 자체가 잘못됨
- ❌ 테스트 없음 + 너무 복잡
- ❌ 점진 불가능 (예: DB 스키마 완전 변경)
- ❌ 기술 스택 완전 deprecated

### ⚠️ 재작성의 함정
**재작성은 항상 예상의 3배 걸림** (Joel Spolsky)
- 1주 예상 → 3주
- 3개월 예상 → 9개월
- 1년 예상 → ... (대부분 실패)

### 권장 의사결정 흐름
```
Q: 테스트가 있나?
  No → 먼저 테스트 작성 (Characterization Tests)
  Yes → 다음

Q: 점진 가능한가?
  Yes → 리팩터 (Strangler Fig + Branch by Abstraction)
  No → 다음

Q: 정말 재작성 외 방법 없나?
  Yes → 재작성 (계획 ×3 시간 예산)
  No → 작은 리팩터 시도
```

### 우리 프로젝트 적용
- `sns-platform/` ES5 → React: **리팩터** (Strangler Fig)
- `ai-writer.html` 옛 단일계정 → 멀티계정: **리팩터** (점진)
- 만약 Supabase → 자체 백엔드: **재작성** (큰 작업, 신중)

## H7. 부채 vs 의도된 단축 구분 (Tech Debt vs Quick Win)

### 부채 (Tech Debt)
- 나중에 갚아야 할 빚 (이자 발생)
- 의식적 선택
- 기한 명시

### 의도된 단축 (Intentional Quick Win)
- MVP, 검증용, 곧 버릴 것
- 갚지 않음
- "이 기능 살아남으면 그때 정리"

### 코드 주석으로 구분
```ts
// 부채: 6/30까지 리팩터 (이슈 #234)
// 이유: 결제 PG 추가 시 추상화 필요
function ugly() { ... }

// 의도된 단축: MVP용, 사용자 검증 후 결정
// 만약 살아남으면 리팩터, 아니면 제거
function temp() { ... }
```

### 결정 기준
| 상황 | 부채 vs 단축 |
|---|---|
| 살아남을 확률 ≥ 80% | 부채 (기한 명시) |
| 50% ~ 80% | 부채 (가벼운) |
| 50% 미만 | 의도된 단축 |
| 실험/A/B 테스트 | 의도된 단축 |

### 우리 프로젝트 사례
- 결제 PG 토스만 임시 (해외 추가 예정): **부채** (기한: 6개월)
- 관리자 수동 충전 처리 (자동화 예정): **부채** (기한: PG 연동 시점)
- AI 모델 카탈로그 하드코딩 (DB 이전 검토 중): **의도된 단축** (사용 패턴 확인 후)
