# ADR-003: Trellis 마이그레이션 범위 — 신규 시작 + 핵심 데이터 선택 이관

## 상태
승인됨 (2026-06-07) — 기획서 §16 D3 권장안 채택, 마스터 지시로 확정.

## 컨텍스트
기반 자산 Asena Trellis(React + Firebase, 글로벌 ~50명 사용)에서 ez-flow(Next.js + Supabase)로
전환한다. Firebase → PostgreSQL은 데이터 모델·인증 체계가 상이하다. 기존 데이터를
전체 이관할지, 신규로 시작하고 핵심만 선택 이관할지 결정이 필요하다.

## 결정
**신규(Supabase)로 시작하고, 핵심 데이터만 선택 이관한다.**
- 이관 대상(우선): 사용자 계정/프로필, 활성 프로젝트, 진행 중 업무.
- 이관 제외(초기): 종료된 프로젝트 이력, 첨부 대용량 아카이브(필요 시 후속 배치).

## 대안
- **옵션 A: 전체 이관** — ❌ 거절.
  - Firebase 문서모델 → 관계형 정규화(기획서 §5) 변환 비용·리스크 큼.
  - 죽은 데이터까지 이관 → 부패 전이.
- **옵션 B(채택): 신규 시작 + 핵심 이관.**

## 트레이드오프
✅ 장점
- 깨끗한 스키마(§5)·RLS(§6) 위에서 시작 → 기술부채 차단.
- 이관 범위 축소 → 일정/리스크 ↓.

❌ 단점
- 일부 과거 이력 즉시 부재(필요 시 후속 이관 작업).

⚠️ 위험
- 이관 매핑 오류 시 데이터 정합성 문제 → **이관 스크립트도 TDD + 검증 카운트(원본 N건 = 대상 N건)** 필수.
- 사용자 인증 전환: Firebase Auth UID ↔ Supabase auth.users 매핑 테이블 필요.

## 결과 (파급)
- Type 1(비가역에 가까움): 이관 스크립트·매핑은 1회성이나 데이터 정합성 영향 큼 → 신중.
- 컷오버 전 양쪽 병행(읽기 검증) 기간 권장.

## 후속 결정
- 이관 컷오버 일정/다운타임 정책.
- 미이관 이력의 보존(Firebase 읽기전용 유지 기간) 또는 후속 배치 이관 여부.
