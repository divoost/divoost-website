# ADR-005: 회의 음성 보존 정책 — 전사 후 즉시 삭제

## 상태
승인됨 (2026-06-07, 보안) — 기획서 §16 D5 + §6.6 권장안 채택, 마스터 지시로 확정.

> ⚠️ 보안 우선 공지(CLAUDE.md I1·I5): 회의 음성은 기밀(전략·인사·거래)을 담을 수 있는
> 고위험 데이터다. 보존할수록 유출 시 피해가 크다. **데이터 흐름·보존을 먼저 결정**한다.

## 컨텍스트
ez-flow는 회의를 녹음 → STT 전사 → AI 요약 → 액션아이템 추출한다(기획서 §8.6, §9.2).
원본 음성 파일(`meetings.audio_path`, Storage)을 얼마나 보관할지 결정해야 한다.

## 결정
**전사(STT) 완료 즉시 원본 음성을 삭제한다.**
- 파이프라인: 업로드 → STT → 전사문/요약 저장(`transcripts`) → **원본 오디오 삭제**.
- 보존하는 것: 텍스트 전사·요약(jsonb)만. 음성 바이너리는 미보존(기본).
- 예외가 필요하면(법적/감사) 별도 명시 동의 + 암호화 + 짧은 보존기간으로 옵트인.

## 대안
- **N일 보관** — ❌ 거절(기본). 유출 시 음성 노출 위험(중). 스토리지 비용·관리 부담.
- **무기한 보관** — ❌ 거절. 보안/프라이버시 위험 최고.

## 트레이드오프
✅ 장점
- 유출 표면 최소화(음성 미보존). 프라이버시·규제 친화.
- Storage 비용 절감.
❌ 단점
- 전사 오류 시 재처리 불가(원본 없음) → STT 실패 시 **삭제 전 사용자 확인/재시도 1회** 후 삭제로 완화.
⚠️ 위험
- 삭제 누락(전사 후 잔존) → Edge Function에서 전사 성공 트랜잭션 내 삭제 보장 + 주기적 스윕(고아 오디오 점검).

## 결과 (파급)
- `ai-meeting-stt` Edge Function이 전사 성공 직후 Storage 객체 삭제 책임.
- 데이터 흐름표(기획서 §6.5) "회의 음성" 행: 보존정책 = **전사 후 삭제**로 확정.
- 버킷은 private + 서명 URL(만료) 기본(기획서 §6.7).

## 후속 결정
- STT 실패 시 사용자 재시도 보존 윈도우(제안: 최대 24시간, 암호화) 확정.
- 법적 보존 의무가 생기는 회의 유형이 있는지(있다면 옵트인 경로) 검토.
